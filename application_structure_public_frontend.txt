JANNATBOOKING PUBLIC FRONTEND APPLICATION STRUCTURE
Audited: 2026-05-26

Purpose
This file documents the current public JannatBooking frontend. It is separate from the PMS/admin hotels_frontend app. It should be kept in sync with future public website, booking, SEO, chat, and SSR migration work.

Runtime
- Directory:
  - D:\JannatBooking\jannatbooking_frontend
- Framework:
  - Create React App.
  - React 18.
  - react-router-dom v5.
  - styled-components.
  - Ant Design.
  - react-helmet.
  - socket.io-client.
  - react-ga4 and react-facebook-pixel.
- Production serving:
  - server.js serves the compiled build directory with Express.
  - package.json build command: npm run build.
  - package.json start command: react-scripts start for local dev.
- Important limitation:
  - CRA renders metadata client-side through react-helmet.
  - Search engines and LLM crawlers that do not execute JavaScript may miss some dynamic page metadata.
  - This is the reason for the future jannatbooking_ssr sibling app.

Public Chat Widget
- Main files:
  - src/Chat/ChatIcon.js
  - src/Chat/ChatWindow.js
  - src/Chat/socket.js
- Visibility:
  - App.js hides ChatIcon when the pathname includes admin or management.
  - This keeps B2C public chat off admin/PMS screens.
- Backend endpoints now used by public chat:
  - POST /api/support-cases/new
  - GET /api/support-cases/client/:id
  - PUT /api/support-cases/client/:id
- Public chat rules:
  - Chat creates client-opened SupportCase records.
  - Public clients can append messages through the client endpoint.
  - Public clients can close/rate the case through the client endpoint.
  - Public clients cannot update support assignment, hotel assignment, AI fields, escalation fields, or admin metadata.
  - If the backend escalates a case, the public client only sees the support conversation; escalation controls/status panels remain admin-only in hotels_frontend.
- Inquiry types in the public form:
  - Room pricing / availability.
  - Bed pricing / availability.
  - Payment Inquiry.
  - Update / questions about reservation.
  - Others.
- Product and AI rules:
  - This is B2C only.
  - B2B/internal chat must remain in hotels_frontend and must not use this widget.
  - Cancellation requests are human-only.
  - Reservation updates are human-reviewed.
  - Public AI assistance, when enabled backend-side, is for pricing by date range, hotels near Al Haram, payment questions, and support triage.

Migration Relationship To jannatbooking_ssr
- jannatbooking_frontend is the current production-safe baseline.
- jannatbooking_ssr should be a sibling public SSR implementation, not a replacement until parity is proven.
- SSR migration must preserve:
  - Public URLs and redirects.
  - Design and content.
  - Hotel search and sorting behavior.
  - Checkout/payment flows.
  - Public B2C chat behavior.
  - Analytics and pixel behavior where legally/configurably appropriate.
- SSR migration must improve:
  - Server-rendered titles/descriptions/canonicals.
  - Dynamic sitemap generation.
  - Dynamic JSON-LD.
  - Crawlability for LLM/search systems.
  - Performance and first-contentful output.

2026-05-27 Public B2C Chat Behavior And Tone Addendum
- Public clients never see admin AI controls.
- The public widget only uses:
  - POST /api/support-cases/new.
  - GET /api/support-cases/client/:id.
  - PUT /api/support-cases/client/:id.
- Chat widget responsibilities:
  - Show AI/staff typing when typing/stopTyping arrives for the active case.
  - Clear typing when receiveMessage arrives.
  - Render Arabic and RTL messages aligned correctly.
  - Render markdown hotel links from AI replies as clickable links that open safely in a new tab.
  - Do not display unreadable legacy text such as "?????" as normal customer-facing content.
- Support tone:
  - Arabic AI replies should address known clients respectfully, for example "أستاذ ناصر".
  - Keep the conversation official, respectful, short, and useful.
  - The bot should never claim to be a human if asked; it is AI-assisted Jannat Booking support monitored by the team.
  - The visible brand remains "Jannat Booking" exactly.
- AI support scope:
  - Helps with hotel options near Al Haram, date-range prices, payment troubleshooting, and reservation triage.
  - Does not cancel, refund, or directly mutate reservations.
  - Reservation updates and cancellations go to a human team member.
- QA notes from 2026-05-27:

  - Arabic payment flow avoided card details and stopped repeating a supplied confirmation number after backend prompt tightening.
  - Human handoff paused AI for reservation update/cancellation-style requests.
  - After QA, backend cleanup removed the temporary Arabic chatbot support cases.
  - No chatbot QA reservations were found; the public chatbot did not create reservations during the test window.
  - Public frontend QA should keep test chat names unmistakable and should not use real customer names in test support cases.
  - Final pre-production cleanup also removed the temporary rude escalation QA case 6a16417650a34b197ce4d2ee.
  - Public chat remains a support/triage surface only; it must not create real reservations during chatbot QA.
